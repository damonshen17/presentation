Stripe Official Best Practices vs Our Architecture

What Stripe recommends, what we do, and the right path forward

1. Stripe's Official Recommendations (from docs)
Dual-Trigger Fulfillment Stripe Docs
From: docs.stripe.com/checkout/fulfillment

"You must use webhooks to make sure
fulfillment happens for every payment."

"Webhooks can sometimes be delayed.
To optimize your payment flow and guarantee
immediate fulfillment when your customer
is present, trigger fulfillment from your
landing page as well."

Pattern: fulfill_checkout(session_id)

  Trigger 1: Webhook
    checkout.session.completed arrives
    --> call fulfill_checkout(session_id)

  Trigger 2: Landing Page
    User lands on success page
    --> extract session_id from URL
    --> call fulfill_checkout(session_id)
    (same function, idempotent)

  Inside fulfill_checkout():
    1. Retrieve session from Stripe API
    2. Check payment_status
    3. Provision access (if not already done)
    4. Mark as fulfilled (idempotent guard)
Store State in Your DB Stripe Docs
From: docs.stripe.com/billing/subscriptions

"Store the status in your database and
check when a user accesses your service.
This approach helps you avoid hitting
rate limits."

"Store customers' credentials and a mapped
timestamp value that represents the access
expiration date for that customer."

"When the customer logs in, you check
whether the timestamp is still in the
future."

Recommended fields to store:
  - subscription.id
  - subscription.status
  - product.id
  - customer.id
  - access expiration date

Key principle:
"The subscription status might change
at any point during its lifetime, even
if your application doesn't directly
make any calls to Stripe."

--> Don't query Stripe on every access check
--> Store state locally, update via webhooks
Webhook-Driven Provisioning Stripe Docs
From: docs.stripe.com/billing/subscriptions/webhooks

"Provision access to your product when
you receive invoice.paid and the
subscription status is active."

"When the subscription changes to
canceled or unpaid, revoke access."

Recommended event --> action mapping:

  invoice.paid
    --> Provision access
    --> Store access expiration date
    "plus a day or two for leeway"

  invoice.payment_failed
    --> Notify customer
    --> Flag account

  customer.subscription.updated
    --> Refresh plan/discount metadata

  customer.subscription.deleted
    --> Revoke access

  customer.subscription.trial_will_end
    --> Verify payment method exists

NOT recommended:
  Querying Stripe API on every page load
  Computing state from raw event data at read time
2. Three Approaches Compared
A. Webhook-Only (Current) Ours
Flow:
  User pays on Stripe
  --> Stripe sends webhooks
  --> Webhooks write to user_subscription +
      stripe_subscription_period
  --> Frontend polls GET /user/profile
  --> get_tier() computes across 5 tables
  --> Returns tier to frontend

Problems:
  1. No fulfillment on landing page
     Stripe explicitly says you need this
  2. State computed at read time
     Stripe says "store the status in your DB"
  3. No Stripe API verification
     Never confirms with Stripe directly
  4. Fallback queries for webhook gaps
     Permanent complexity for transient problem
  5. No retry for failed events
     Failed webhooks = permanent desync

Verdict:
  Violates Stripe's dual-trigger pattern
  Violates "store status in your DB"
  Over-engineers the read path
  Under-engineers the write path
B. API-Call-First + Webhook Correction Your Idea
Flow:
  User pays on Stripe
  --> Stripe redirects to success page
  --> Success page calls Stripe API
      to retrieve session + subscription
  --> Immediately provisions access
  --> Webhooks arrive later
  --> Webhooks CORRECT/CONFIRM state

Pros:
  Instant provisioning (no webhook wait)
  Matches Stripe's dual-trigger pattern!
  Webhook becomes safety net, not sole path

Cons:
  Only works when user visits success page
  Doesn't handle renewals, cancellations
  (those happen without user being present)
  Still need webhooks for ongoing lifecycle

Verdict:
  Good for initial checkout fulfillment
  Incomplete for full subscription lifecycle
  Need to combine with materialized state
C. Stripe's Full Pattern Best Practice
Flow:
  CHECKOUT:
    User pays on Stripe
    --> Success page: retrieve session from
        Stripe API, provision immediately
    --> Webhook: same fulfill function
        (idempotent, runs only once)

  ONGOING LIFECYCLE:
    Webhooks update materialized state:
    --> invoice.paid: set tier=PRO,
        access_expiry = period_end + 2 days
    --> invoice.payment_failed: set
        subscription_state = PAYMENT_FAILED
    --> subscription.deleted: set tier=FREE

  ACCESS CHECK:
    Read stored tier/expiry from user record
    No computation, no JOINs, no fallbacks

  SAFETY NET:
    Nightly reconciliation with Stripe API
    Catches any drift from missed webhooks

Verdict:
  Matches ALL Stripe recommendations
  Instant checkout + reliable lifecycle
  Simple reads, smart writes
  Self-healing via reconciliation
3. The Right Answer: Combine Your Idea + Materialized State

Your intuition is exactly right and aligns with Stripe's official recommendation. The best practice is a dual-trigger + materialized state pattern. Neither webhook-only nor API-call-only is sufficient alone.

THE COMPLETE PATTERN (Stripe-recommended + your idea combined):

=====================================================
 LAYER 1: Checkout Fulfillment (Dual-Trigger)        
=====================================================

  Trigger A: Success Page (instant, user is present)

    User lands on /checkout-success?session_id=cs_xxx
                          |
                          v
    Backend: fulfill_checkout(session_id)
      1. stripe.checkout.Session.retrieve(session_id, expand=['subscription'])
      2. if session.payment_status == 'paid':
      3.    subscription = session.subscription
      4.    UPDATE user_info SET
      5.      tier = derive_tier(subscription.plan),
      6.      subscription_state = subscription.status,
      7.      fulfilled_session_id = session_id  -- idempotency guard
      8.    Create user_subscription record
      9.    Return tier=PRO to frontend
      User sees PRO immediately. No polling needed!

  Trigger B: Webhook (reliable, covers all cases)

    checkout.session.completed arrives
                          |
                          v
    Backend: fulfill_checkout(session_id)  -- SAME function
      1. Check: if already fulfilled (fulfilled_session_id exists) --> skip
      2. Otherwise: same logic as Trigger A
      Idempotent. If success page already handled it, this is a no-op.
      If user closed browser before success page, this catches it.

=====================================================
 LAYER 2: Ongoing Lifecycle (Webhook + Materialize)   
=====================================================

  invoice.paid webhook:
    UPDATE user_info SET
      ever_paid = TRUE (if not trial $0 invoice),
      access_expiry = period_end_time + 2 days
    + Upsert stripe_subscription_period (keep for audit)

  invoice.payment_failed webhook:
    UPDATE user_info SET subscription_state = 'payment_failed'
    + Upsert stripe_subscription_period (keep for audit)

  customer.subscription.updated webhook:
    UPDATE user_info SET
      tier = derive_tier(new_plan),
      subscription_state = new_status
    + Update user_subscription record (keep for history)

  customer.subscription.deleted webhook:
    UPDATE user_info SET
      tier = FREE (if no other active sub),
      subscription_state = 'inactive'
    + Update user_subscription record

=====================================================
 LAYER 3: Access Check (Simple Read)                  
=====================================================

  GET /user/profile:
    SELECT tier, ever_paid, subscription_state, access_expiry
    FROM user_info WHERE user_hash = ?
    -- 1ms, indexed, no JOINs, no fallbacks

  Optional bonus: if access_expiry < now AND tier != FREE:
    -- Grace period expired, subscription may have lapsed
    -- Could trigger async Stripe API check

=====================================================
 LAYER 4: Safety Net (Reconciliation)                 
=====================================================

  Nightly cron:
    For users with tier=PRO:
      Verify active Stripe subscription exists
      If not: downgrade to FREE, alert ops
    For users with tier=FREE:
      Check if Stripe has active subscription we missed
      If yes: upgrade to PRO, alert ops
    -- Catches missed webhooks, manual Stripe changes, bugs

5. Direct Answer to Your Question
Q: Should we sync state after our API call + use webhook to correct later?

A: YES. This is exactly what Stripe recommends.

Stripe calls this the "dual-trigger fulfillment" pattern:

  "Webhooks can sometimes be delayed. To optimize your payment flow
  and guarantee immediate fulfillment when your customer is present,
  trigger fulfillment from your landing page as well."

Q: Is our current approach (webhook-only + compute at read time) best practice?

A: NO. It violates multiple Stripe recommendations:

  1. No landing page fulfillment
     Stripe: "trigger fulfillment from your landing page as well"
     Ours: webhook-only, frontend polls hoping webhook arrived

  2. State computed at read time instead of stored
     Stripe: "Store the status in your database and check when a user accesses your service"
     Ours: 4-query UNION ALL + 6 subqueries computed on every request

  3. No access expiration date
     Stripe: "store a mapped timestamp value that represents the access expiration date"
     Ours: compute from period_start_time/period_end_time/charging_time JOINs

  4. No reconciliation
     Stripe: implied by "subscription status might change at any point"
     Ours: no safety net for missed webhooks

Q: What's the community consensus?

The overwhelming pattern across Stripe integrations:
  1. Fulfill on success page (Stripe API call, instant for user)
  2. Webhooks update materialized state (reliable backend)
  3. Simple access checks (read a column, not compute a query)
  4. Reconciliation as safety net (nightly or hourly)

Our current architecture inverted this:
  - Write path is minimal (just save webhook data)
  - Read path does all the work (compute state from raw data)
  - This is backwards. Writes are rare, reads are constant.
  - Do the work ONCE at write time, not N times at read time.
