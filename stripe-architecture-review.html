為什麼我們的 Stripe 訂閱架構需要重構

從 Stripe 官方最佳實踐出發，分析現有架構的問題，以及為什麼「寫入時計算狀態」比「讀取時計算狀態」更好

TL;DR

我們目前的架構在每次 API 請求時，都用 4 個 UNION ALL 查詢跨 5 張表即時計算使用者的 tier（免費/付費）。這不是 Stripe 推薦的做法。Stripe 建議：在 webhook 處理時把狀態寫入資料庫，讀取時直接讀欄位。同時，結帳完成後應該在 success page 直接呼叫 Stripe API 確認狀態，而不是靠 polling 等 webhook。這個架構問題也導致了生產環境中的「試用期已結束」競態條件錯誤（詳見第 7 節）。

1 現在的架構長什麼樣

當使用者打開任何頁面，前端都會呼叫 GET /user/profile 來取得使用者資訊。這個 API 需要回傳幾個關鍵欄位：

欄位意義怎麼取得
tier使用者等級（free / pro）即時計算：4 個 UNION ALL 查詢
everPaid是否曾經付過款即時計算：6 個子查詢
subscriptionState訂閱狀態即時計算：3 分支 CASE 查詢
userTrialAvailable是否可以試用即時計算：查詢歷史訂閱

這些欄位全部都不是從資料庫直接讀的，而是每次請求都跨 5 張表即時計算。

每次 GET /user/profile 的執行流程：

  前端請求 /user/profile
       │
       ▼
  get_tier(user_hash)                        ~150ms
    ├── Q1: user_subscription JOIN stripe_subscription_period（個人訂閱 + 有效帳單期間）
    ├── Q2: team_member JOIN team JOIN user_subscription JOIN stripe_subscription_period（團隊訂閱）
    ├── Q3: user_subscription 降級查詢（webhook 還沒到的 fallback）
    └── Q4: team 降級查詢（同上）
    UNION ALL 合併 → 有任何一個命中 → PRO，否則 FREE
       │
       ▼
  has_ever_paid(user_hash)                    ~100ms
    ├── 個人訂閱：JOIN stripe_subscription_period，排除 trial_extension
    ├── 團隊訂閱：JOIN team_member + stripe_subscription_period
    ├── 個人 fallback（stripe_subscription_period 還沒建立時）
    └── 團隊 fallback（同上）
       │
       ▼
  get_user_subscription_state(user_hash)      ~80ms
    ├── CASE WHEN: 有帳單期間 + charging_time IS NULL → PAYMENT_FAILED
    ├── CASE WHEN: 有帳單期間 + charging_time IS NOT NULL → ACTIVE
    └── CASE WHEN: fallback 只看 gw_subscription_status → ACTIVE
       │
       ▼
  回傳 { tier, everPaid, subscriptionState, ... }

  總計：~350ms+ 的 SQL 查詢，每次頁面載入都要執行

注意：user_info 表上其實已經有 tier 和 ever_paid_for_pro 這兩個欄位，但它們是「死程式碼」——有欄位卻從不寫入，程式碼裡標記了 TODO: Remove。

2 為什麼會變成這樣：Webhook 依賴鏈問題

要理解為什麼 get_tier() 需要這麼複雜，必須先理解 Stripe 的 webhook 事件是怎麼到達的。

Stripe 事件到達順序

當使用者完成結帳後，Stripe 會依序但不保證順序地發送多個事件：

使用者按下「訂閱」後，Stripe 發送的事件：

  T+0s   checkout.session.completed
           → 只更新 stripe_checkout_session 的狀態
           → 此時沒有任何訂閱資料！tier 還是 FREE

  T+2s   customer.subscription.created
           → 建立 user_subscription（plan、status、effective_time）
           → 但還沒有 stripe_subscription_period！
           → Q1/Q2 需要的 charging_time 還不存在

  T+5s   invoice.payment_succeeded
           → 建立 stripe_subscription_period（charging_time、period_start/end）
           → 現在 Q1/Q2 終於可以判斷 tier = PRO 了

問題來了：get_tier() 需要同時用到 user_subscription（來自事件 B）和 stripe_subscription_period（來自事件 C）的資料。但這兩個事件在不同時間到達。

Fallback 查詢的由來

因為事件 B 和 C 之間有時間差，所以工程師加了 fallback 查詢（Q3/Q4）：

問題：Fallback 查詢是永久的，但它解決的是暫時的問題

Q3/Q4 的存在是為了處理「subscription.created 已到但 invoice.payment_succeeded 還沒到」的幾秒鐘窗口期。但這個 fallback 是永久啟用的，它的判斷條件比 Q1/Q2 更寬鬆（不檢查 charging_time），所以在某些邊界情況下會返回錯誤結果。

PR #1064：一個典型的邊界 Bug

這個 PR 修的就是 has_ever_paid() 的 fallback 邏輯錯誤。問題在於同一個 gw_subscription_id 在 trial → active 轉換時會被重用，導致 trial 期間的 stripe_subscription_period 記錄和付費期間的記錄混在一起。

修復方式是加上 period_start_time >= effective_time 的過濾條件。但這只是在已經很複雜的查詢上再疊加一層複雜度。根本原因是：我們在讀取時才去分辨「這是 trial 還是付費」，而 webhook handler 在處理事件的當下其實已經知道答案。

3 Stripe 官方怎麼說

查閱 Stripe 官方文件後，我們發現我們的做法違反了三個核心建議：

建議一：雙觸發履行（Dual-Trigger Fulfillment）

"Webhooks can sometimes be delayed. To optimize your payment flow and guarantee immediate fulfillment when your customer is present, trigger fulfillment from your landing page as well."

— docs.stripe.com/checkout/fulfillment

Stripe 建議寫一個 fulfill_checkout(session_id) 函式，然後在兩個地方呼叫它：

  1. Success page（使用者結帳完回到網站時）：直接呼叫 Stripe API 取得 session 和 subscription 狀態，立即開通權限。
  2. Webhook（checkout.session.completed）：同一個函式，冪等設計（如果 success page 已經處理過就跳過）。

我們目前只有 webhook 這條路，前端靠 polling（每秒一次，最多 10 次）來等 webhook 處理完。如果 webhook 延遲超過 10 秒，使用者就會看到錯誤的 FREE 狀態。

建議二：把狀態存到你的資料庫

"Store the status in your database and check when a user accesses your service. This approach helps you avoid hitting rate limits."

— docs.stripe.com/billing/subscriptions

"Store customers' credentials and a mapped timestamp value that represents the access expiration date for that customer on your site. When the customer logs in, you check whether the timestamp is still in the future."

— docs.stripe.com/billing/subscriptions/webhooks

Stripe 明確說：收到 webhook 時就把狀態寫進你的資料庫，使用者訪問時直接讀欄位。不要每次都去重新計算或查詢 Stripe API。

我們目前的做法恰恰相反：webhook 只寫入原始資料（user_subscription、stripe_subscription_period），然後每次讀取時用 ~490 行的 SQL 去即時推算狀態。

建議三：Webhook 驅動的權限變更

"Provision access to your product when you receive invoice.paid and the subscription status is active. When the subscription changes to canceled or unpaid, revoke access to your product."

— docs.stripe.com/billing/subscriptions/webhooks

Stripe 的模式是：

  • invoice.paid → 開通權限，記錄 access expiration date
  • invoice.payment_failed → 標記帳號、通知使用者
  • subscription.deleted → 撤銷權限

注意：Stripe 說的是「開通/撤銷權限」，也就是改變你資料庫裡的狀態，而不是「記錄事件原始資料然後讓讀取端自己算」。

4 核心問題：讀取時計算 vs 寫入時計算

我們的架構問題可以用一句話總結：

我們把工作放在了錯誤的地方

寫入（webhook）很少發生（使用者訂閱、續費、取消——一個月可能就幾次），但讀取（API 請求）不斷發生（每次頁面載入、每次 polling）。我們把計算邏輯放在讀取端，等於是把少量事件的複雜性分攤到了大量請求上。

❌ 現在：讀取時計算（Read-Heavy）

寫入路徑（webhook）：

  • 把原始資料存到 user_subscription
  • 把帳單期間存到 stripe_subscription_period
  • 完成。不做任何狀態推算。

讀取路徑（每次 API 請求）：

  • 4 個 UNION ALL 跨 5 張表算 tier
  • 6 個子查詢算 ever_paid
  • 3 分支 CASE 算 subscription_state
  • ~350ms，~490 行 SQL

✅ 應該：寫入時計算（Write-Heavy）

寫入路徑（webhook）：

  • 原有邏輯不變（寫 user_subscription 等）
  • 新增：UPDATE user_info SET tier, ever_paid, subscription_state
  • 每個 handler 多 ~5 行程式碼

讀取路徑（每次 API 請求）：

  • SELECT tier, ever_paid, subscription_state FROM user_info
  • ~1ms，索引主鍵查詢
  • 不需要 JOIN、不需要 fallback

為什麼 Stripe 認為寫入時計算更好？

有三個原因：

原因 1：寫入時擁有完整上下文

當 invoice.payment_succeeded webhook 到達時，handler 已經知道：這是 trial 的 $0 invoice 還是真正的付費、billing_reason 是 subscription_create 還是 subscription_cycle、plan 是什麼、recurrence 是什麼。所有判斷條件在寫入當下都是已知的，不需要事後再去推算。

而讀取時計算（如 has_ever_paid()）必須從冰冷的資料庫記錄中反向推理這些答案，所以才需要 trial_extension 排除邏輯、period_start_time >= effective_time 等複雜條件。

原因 2：寫少讀多

一個使用者的訂閱狀態變化（建立、續費、升級、取消）一個月可能只發生 1-2 次。但 GET /user/profile 在每次頁面載入、每次 polling、每次路由切換時都會被呼叫。在寫入時多做 5 行工作，可以省下讀取時 350ms × N 次的計算。

原因 3：消除 Fallback 的必要性

讀取時計算之所以需要 fallback，是因為不同 webhook 事件在不同時間寫入不同的表。讀取端不知道「到底是 webhook 還沒到，還是這個使用者真的不是 PRO」。

但如果在寫入時就更新 user_info.tier，那麼：

  • subscription.created handler 寫入 tier = PRO → 讀取端立刻看到 PRO
  • 不需要等 invoice.payment_succeeded 才能確認
  • 不需要 fallback 查詢
  • 不需要 trial/active 的區分邏輯（handler 已經知道）

5 完整的修復方案

結合 Stripe 官方建議和我們的實際情況，推薦的架構有四層：

第一層：結帳履行（Dual-Trigger）

新增 fulfill_checkout(session_id) 函式：

  觸發點 A：Success Page（使用者看得到，即時）
    使用者結帳完 → 跳轉到 /checkout-success?session_id=cs_xxx
    → 後端呼叫 stripe.checkout.Session.retrieve(session_id)
    → 取得 subscription 狀態
    → 寫入 user_info: tier=PRO, subscription_state=ACTIVE
    → 建立 user_subscription 記錄
    → 回傳給前端 → 使用者立刻看到 PRO，不需要 polling！

  觸發點 B：Webhook（可靠，涵蓋所有情況）
    checkout.session.completed 事件到達
    → 呼叫同一個 fulfill_checkout(session_id)
    → 檢查：如果已經處理過 → 跳過（冪等）
    → 如果還沒處理 → 執行同樣的邏輯
    → 覆蓋使用者沒有訪問 success page 的情況（關掉瀏覽器等）

這正是 Stripe 說的："trigger fulfillment from your landing page as well"。兩條路呼叫同一個冪等函式，誰先到誰處理。

第二層：訂閱生命週期（Webhook 物化狀態）

結帳之後的所有狀態變更（續費、付款失敗、取消、升降級）都發生在使用者不在場的時候，只能靠 webhook。但 webhook handler 處理完原有邏輯後，多做一步：更新 user_info。

Webhook 事件現有邏輯（保留）新增：更新 user_info
subscription.created 建立 user_subscription SET tier=PRO, subscription_state=status
subscription.updated 過期舊紀錄、建立新 user_subscription SET tier=derive(plan), subscription_state=new_status
subscription.deleted 設定 cancel_at、expiry_time SET tier=FREE（如無其他活躍訂閱）, subscription_state=INACTIVE
invoice.paid Upsert stripe_subscription_period SET ever_paid=TRUE（如非 trial $0 invoice）
invoice.payment_failed Upsert stripe_subscription_period SET subscription_state=PAYMENT_FAILED

每個 handler 只需要多加 ~5 行程式碼。現有的原始資料寫入完全保留（audit trail）。

第三層：存取檢查（簡單讀取）

修改前（現在）：
  GET /user/profile
    → get_tier()                       4 UNION ALL, 5 表 JOIN   ~150ms
    → has_ever_paid()                  6 子查詢, trial 排除     ~100ms
    → get_user_subscription_state()    3 分支 CASE + fallback   ~80ms
    總計 ~350ms+，~490 行 SQL 邏輯

修改後（目標）：
  GET /user/profile
    → SELECT tier, ever_paid, subscription_state FROM user_info WHERE user_hash = ?
    總計 ~1ms，索引主鍵查詢，10 行程式碼

第四層：安全網（定期校正）

任何物化狀態都可能因為 webhook 失敗、bug、或 Stripe 手動操作而產生偏差。所以需要一個每日校正任務：

  • 對所有 tier=PRO 的使用者，呼叫 Stripe API 確認確實有活躍訂閱
  • 對所有 tier=FREE 的使用者，檢查是否有我們漏掉的活躍訂閱
  • 發現不一致時：修正資料庫 + 發送告警

6 遷移策略（零停機）

這個重構可以分 5 個階段安全執行，不需要一次性切換：

Phase 1：雙寫（Dual Write）

Webhook handler 開始寫入 user_info.tier 等欄位，但讀取路徑不變，還是用舊的 get_tier()。風險為零：寫入是附加的，讀取不受影響。

Phase 2：回填（Backfill）

跑一次腳本，對所有使用者呼叫現有的 get_tier() 和 has_ever_paid()，把結果寫入 user_info 欄位。確保所有歷史使用者都有正確的物化狀態。

Phase 3：影子讀取（Shadow Read）

讀取路徑同時呼叫新舊兩種方式，比較結果。如果不一致，記錄到監控系統。跑 1-2 週確認新舊一致。

Phase 4：切換讀取

讀取路徑改為直接讀 user_info 欄位。即時省下 ~350ms/request。

Phase 5：清理

刪除 get_tier()（150 行）、has_ever_paid()（130 行）、get_user_subscription_state()（130 行）等。移除 fallback 查詢。移除前端的 useSubscriptionStatePolling。刪除 get_tier 的索引遷移。總共刪除 ~490 行複雜 SQL 邏輯。

7 實際案例：「試用期已結束」錯誤

我們在生產環境中觀察到多起錯誤訊息：

錯誤訊息

"Your trial period has already ended or your subscription is not in trial status. No action is needed."

來源：payment/router.py 的 POST /subscription/end-trial endpoint

這個 endpoint 的邏輯是：

  1. 呼叫 get_user_active_subscription() 查詢最新的訂閱
  2. 檢查 gw_subscription_status == 'trialing'
  3. 如果不是 trialing → 拋出錯誤
  4. 呼叫 Stripe API 設定 trial_end = now + 1s

根本原因是：Trial 的到期時間由 Stripe 控制，我們的 FE/BE 都不知道確切的到期時刻。我們只能靠 webhook 得知 trial 已結束，但 webhook 有延遲，導致 DB 狀態與 Stripe 實際狀態不同步。

核心問題：我們不知道 Trial 何時結束

Trial 到期是 Stripe 的事，不是我們的事

當使用者開始試用時，Stripe 設定了一個 trial_end 時間戳。到期時，Stripe 自動將 subscription status 從 trialing 改為 active，然後發送 subscription.updated webhook 通知我們。

問題是：我們的前端和後端都不知道這個轉換何時發生。我們唯一的信息來源就是 webhook——但 webhook 可能延遲數秒甚至數分鐘。

情境一：Trial 自然到期，Webhook 已到（最常見）

使用者看到過時的 UI：

  T+0      使用者開始試用
           subscription.created → gw_subscription_status = 'trialing'

  T+7天    Trial 到期（由 Stripe 控制，我們的 FE/BE 不知道）
           Stripe 自動將 subscription status 改為 active
           Stripe 發送 subscription.updated webhook

  T+7天    Webhook 到達 → handle_customer_subscription_updated()
    +2s      ├── 過期舊 user_subscription（trialing）
             └── 建立新 user_subscription（status = 'active'）

  T+7天    使用者開啟頁面（或頁面還停留在舊狀態）
    +10s     → 前端顯示「結束試用」按鈕（基於上次取得的 profile，此時已過時）
             → 使用者點擊「結束試用」
             → end_trial 讀取 DB: gw_subscription_status = 'active'
             → 拋出錯誤：「試用期已結束」

  根本原因：前端不知道 trial 已自然到期。
  Webhook 已更新 DB，但前端還在顯示過時的 UI。

情境二：Trial 到期但 Webhook 延遲

DB 狀態落後於 Stripe 實際狀態：

  T+7天    Trial 到期（Stripe 側已經是 active）
    +0s      但 webhook 還沒到達
             DB 中 gw_subscription_status 還是 'trialing'（過時！）

  T+7天    使用者點擊「結束試用」
    +1s      → end_trial 讀取 DB: gw_subscription_status = 'trialing' ✓ 通過本地檢查
             → 呼叫 Stripe API: trial_end = now + 1s
             → Stripe 回傳錯誤：subscription 已經不是 trialing
             → 或者 Stripe 接受了（因為剛剛才轉換），但效果不確定

  T+7天    Webhook 終於到達
    +5s      → 更新 DB 為 active
             → DB 狀態和之前 end_trial 的操作可能衝突

  根本原因：我們的 DB 比 Stripe 慢了好幾秒。
  本地狀態檢查通過了，但 Stripe 上的真實狀態已經不同。

情境三：使用者在 Trial 到期邊界操作

時間差造成的模糊地帶：

  T+7天    使用者在 trial 到期前幾秒點擊「結束試用」
    -3s      → end_trial 讀取 DB: gw_subscription_status = 'trialing' ✓
             → 呼叫 Stripe API: trial_end = now + 1s

  T+7天    就在 API 呼叫的同時，Stripe 自動讓 trial 到期
    +0s      → Stripe 發送 subscription.updated（trialing → active）

  T+7天    兩件事同時發生：
    +1s      A) 我們的 trial_end = now + 1s 請求到達 Stripe
             B) Stripe 的自動到期 webhook 到達我們
             → 結果取決於哪個先被處理，行為不確定

  根本原因：Trial 到期是 Stripe 控制的時間點，
  我們無法預測它和使用者操作之間的先後順序。

為什麼這是架構的根本問題

我們完全依賴 Webhook 來同步 Trial 狀態

問題的核心是：Trial 何時到期是 Stripe 決定的，我們的 FE/BE 都無法提前知道。我們只能被動等待 subscription.updated webhook 到達後才更新 DB。

這產生了一個不可避免的時間窗口：Stripe 上的 subscription 已經從 trialing 變成 active，但我們的 DB 還沒收到通知。在這個窗口內：

  • 前端 UI 顯示過時的 trialing 狀態（因為 GET /user/profile 讀到舊資料）
  • 後端的 gw_subscription_status 檢查基於過時的 DB 狀態
  • 使用者的操作（點擊「結束試用」）基於錯誤的前提

這跟 get_tier() 和 has_ever_paid() 的問題如出一轍：我們的系統依賴 webhook 來同步狀態，但 webhook 有延遲，而我們沒有任何機制來處理這個延遲。

新架構如何解決

不信任本地 DB，直接問 Stripe

在新架構下，end_trial endpoint 會：

  1. 跳過本地狀態檢查：不讀 DB 的 gw_subscription_status
  2. 直接呼叫 Stripe API 結束試用 → Stripe 是 trial 狀態的唯一真相來源（Single Source of Truth）
  3. 如果 Stripe 回傳成功 → 基於回傳值寫入 user_info（tier、subscription_state）
  4. 如果 Stripe 回傳「不是 trialing」→ 表示 trial 已自然到期 → 回傳成功（使用者的意圖已達成）
  5. 後到的 webhook 做冪等檢查 → 已處理過就跳過

關鍵改變：讓 Stripe 而非我們的 DB 來判斷 trial 是否還在進行中。Stripe 是管理 trial 的那一方，它永遠知道正確答案。

短期修復（在全面重構之前）

三個可立即實施的改善

  1. 移除本地前置檢查：不在 DB 檢查 trialing 狀態，直接呼叫 Stripe API。Stripe 知道 trial 是否還在，我們不需要猜。如果 Stripe 說不行，再回傳對應的錯誤給使用者
  2. 冪等處理：如果 Stripe 回傳訂閱已經是 active（trial 已自然到期），回傳成功而非錯誤——使用者的意圖是「開始付費」，而這已經發生了
  3. 前端即時狀態：在 end_trial 頁面載入時重新取得 /user/profile，如果訂閱已不是 trialing 就隱藏「結束試用」按鈕，減少使用者看到過時 UI 的機會

8 常見疑問：這樣不是要打很多 Stripe API？

一個常見的疑問是：「如果不信任本地 DB，改成直接問 Stripe，不是要打很多 API 嗎？」

答案是：不會。因為讀取和寫入是完全不同的場景。

讀取（高頻）→ 讀 DB，不碰 Stripe

每次頁面載入的 GET /user/profile 讀的是 user_info 表的物化欄位。這是前面說的 write-time materialization：webhook 到達時寫入 DB，讀取時直接 SELECT，Stripe API 呼叫次數 = 0。

寫入（低頻）→ 本來就要呼叫 Stripe

像 end-trial、結帳這些使用者操作，本來就需要呼叫 Stripe API。改的只是移除前面不可靠的 DB 前置檢查，Stripe API 呼叫次數不變（甚至更少）。

end_trial 的 Stripe API 呼叫次數比較：

  現在：
    1. 讀 DB 檢查 gw_subscription_status == 'trialing'     ← 可能過時
    2. 呼叫 Stripe API: subscriptions.modify()              ← 1 次 API call
    如果步驟 1 失敗（trial 已到期）→ 直接拋錯，不呼叫 Stripe
    問題：本該成功的操作被過時的 DB 狀態擋掉了

  改後：
    1. 直接呼叫 Stripe API: subscriptions.modify()          ← 1 次 API call（一樣）
    2. 根據 Stripe 回傳值更新 DB
    Stripe 是 trial 狀態的 source of truth，由它來判斷

各場景的 API 呼叫量對比

場景頻率現在的 Stripe API 呼叫改後的 Stripe API 呼叫
頁面載入（讀 tier） 極高（每次載入） 0 次（但 350ms SQL） 0 次（1ms DB 讀取）
結帳完成 低（每次購買） 0 次（等 webhook + 10s polling） +1 次 session.retrieve（換來即時開通）
End trial 低（每人最多一次） 1 次 subscriptions.modify 1 次 subscriptions.modify（一樣）
續費 / 取消 / 付款失敗 低（webhook 驅動） 0 次 0 次（webhook 寫入 + 更新 user_info）
每日校正 1 次/天（離線） 不存在 批次 list subscriptions（離線，不在請求路徑上）

Stripe Rate Limit：完全不是問題

Stripe live mode 的 rate limit 是 100 req/s。我們的寫入操作（結帳、end-trial、取消）一天可能就幾百次，遠低於限制。唯一高頻的場景是頁面載入讀 tier，而那完全不呼叫 Stripe——改前改後都是讀自己的 DB。

唯一新增的 Stripe API 呼叫是：

  • Success page：多一次 checkout.session.retrieve()，但這換來的是消除 10 秒 polling 和即時開通體驗
  • 每日校正：離線批次跑，用 subscriptions.list 拉資料，不在使用者請求路徑上

一句話總結

Webhook 處理事件的當下擁有最完整的上下文——它知道這是 trial 還是付費、是建立還是取消、是個人還是團隊。在那個時刻把狀態寫死，比事後讓每個讀取請求從碎片化的資料中反向推理，要簡單、快速、正確得多。使用者操作（如 end-trial）直接問 Stripe 要答案，不信任可能過時的本地 DB。高頻讀取走 DB，低頻寫入走 Stripe API——API 呼叫量基本不變，但正確性和速度大幅提升。這也是 Stripe 推薦的做法。
