Webhook Dependency Problem & Simplified Architecture

Why get_tier, has_ever_paid, and get_user_subscription_state are complex -- and how to fix it

1. The Root Cause: Webhook Event Dependency Chain
STRIPE FIRES THESE EVENTS (in roughly this order, but NOT guaranteed):

  checkout.session.completed  ------>  Updates: stripe_checkout_session state=COMPLETE
          |                                // No subscription data yet!
          v
  customer.subscription.created --->  Creates: user_subscription (status=trialing|active)
          |                                // Has: plan, recurrence, gw_subscription_status
          |                                // Missing: NO stripe_subscription_period yet!
          v
  invoice.payment_succeeded  ------>  Creates: stripe_subscription_period (charging_time set)
                                           // NOW we have all the data needed for tier

THE PROBLEM: Each query needs data from MULTIPLE webhook events that arrive at DIFFERENT times.

  get_tier() needs:
    - user_subscription.gw_subscription_status   <-- from subscription.created
    - stripe_subscription_period.charging_time   <-- from invoice.payment_succeeded
    - stripe_subscription_period.period_end_time <-- from invoice.payment_succeeded
    Gap: subscription.created arrives but invoice.payment_succeeded hasn't yet
    Solution: Fallback queries Q3/Q4 that trust gw_subscription_status alone

  has_ever_paid() needs:
    - user_subscription.gw_subscription_status   <-- from subscription.created
    - stripe_subscription_period.charging_time   <-- from invoice.payment_succeeded
    - user_subscription.effective_time           <-- from subscription.created/updated
    Gap: same as above, PLUS trial-to-active reuses same subscription_id
    Solution: Fallback queries + trial_extension exclusion + period_start >= effective_time (PR #1064 fix!)

  get_user_subscription_state() needs:
    - user_subscription existence + status        <-- from subscription.created
    - stripe_subscription_period.charging_time   <-- from invoice.payment_succeeded
    - team.deactivated_at                        <-- from subscription.deleted
    Gap: same dependency on period record existing
    Solution: Fallback to gw_subscription_status when no period exists
get_tier() Has the SAME Problem Yes
get_tier() vulnerability analysis:

Q1 (Primary): Personal sub + valid period
  Requires: stripe_subscription_period
    - charging_time IS NOT NULL
    - period_start_time <= now <= period_end_time
    - Joins on gw_subscription_id
  Fails when: invoice.payment_succeeded hasn't arrived yet

Q2 (Primary): Team sub + valid period
  Same as Q1 but adds: team_member + team JOINs
  Fails when: same + team membership timing edge cases

Q3 (Fallback): Personal sub without period
  Trusts: gw_subscription_status == ACTIVE
  No period join needed
  BUT: doesn't check charging_time, so TRIALING users
  who got subscription.created with status=active
  would pass even without payment (trial upgrade edge case)

Q4 (Fallback): Team sub without period
  Same as Q3 for teams

KEY ISSUE: Q3/Q4 are LESS strict than Q1/Q2
They exist ONLY to cover webhook delays but are
permanently enabled, creating a weaker code path
that could return PRO for edge cases that Q1/Q2
would correctly reject.

2. Webhook Timeline: Where Each Piece of Data Becomes Available
TIMELINE: New Pro Monthly Subscription (with trial)

T=0s   User clicks "Start Free Trial" on learnext
       Frontend calls POST /checkout-session
       Backend creates Stripe checkout session
       Saves: stripe_checkout_session (state=OPEN)

T=30s  User completes payment on Stripe page
       Stripe redirects to /checkout-success
       Frontend starts polling (1s x 10 = 10s window)

T=31s  checkout.session.completed webhook arrives
       Updates: stripe_checkout_session state=COMPLETE
       tier=FREE, ever_paid=false, sub_state=never_subscribed (no change yet)

T=32s  customer.subscription.created webhook arrives
       Creates: user_subscription
         gw_subscription_status = 'trialing'
         plan = 'pro', recurrence = 'month'
         effective_time = now
       tier=PRO (via fallback Q3: status=trialing passes... wait, no!
       Q3 requires status=ACTIVE, but trialing != active
       tier=FREE! User still sees FREE tier during trial start!

T=33s  invoice.payment_succeeded webhook arrives (trial invoice, $0)
       Creates: stripe_subscription_period
         charging_time = now (even for $0 trial invoice!)
         period_start_time / period_end_time set
       NOW: Q1 can match: period exists + charging_time NOT NULL
       tier=PRO, sub_state=ACTIVE

T=40s  Frontend polling has ended (10s window closed at T=40s)
       If webhook T=33 arrived after T=40, user sees FREE until manual refresh

TIMELINE: Trial Ends, Becomes Paid (month-to-year upgrade within trial)

T+7d   customer.subscription.updated webhook arrives
         previous_status = 'trialing', new_status = 'active'
       Handler:
         1. Expires old user_subscription (expiry_time = now)
         2. Creates NEW user_subscription (status=active, new effective_time)
       Problem: Same gw_subscription_id now has TWO user_subscription rows
       AND stripe_subscription_period records from BOTH trial and active periods

       has_ever_paid() must now distinguish:
         - Trial period SSP (charging_time set, but was $0 trial)
         - Active period SSP (charging_time set, actual payment)
       This is exactly what PR #1064 fixed: period_start_time >= effective_time

TIMELINE: Payment Fails on Renewal

T+30d  invoice.payment_failed webhook arrives
       Upserts: stripe_subscription_period
         charging_time = NULL, payment_failed_count++
       get_user_subscription_state detects: period exists + charging_time IS NULL = PAYMENT_FAILED

T+30d  customer.subscription.updated webhook arrives
         new_status = 'past_due'
       Updates: user_subscription.gw_subscription_status = 'past_due'

       Race: if subscription.updated arrives BEFORE invoice.payment_failed:
         Q1 still shows ACTIVE (old period still valid with charging_time)
         Only after invoice.payment_failed creates new period does state flip
3. Simplified Architecture: Write State at Webhook Time
Current: Compute at Read Time Complex
WRITE PATH (webhook handlers):
  subscription.created --> user_subscription
  invoice.payment_succeeded --> stripe_subscription_period
  subscription.updated --> user_subscription (new row)
  subscription.deleted --> user_subscription (cancel_at)
  // State is scattered across tables
  // No single source of truth

READ PATH (every API request):
  GET /user/profile -->
    get_tier()                    ~150ms  4 UNION ALL
    has_ever_paid()               ~100ms  6 subqueries
    get_user_subscription_state() ~80ms   3-branch CASE
    user_eligible_to_trial()      ~20ms
    N x get_user_subscription_state() per team
  Total: ~350ms+ of SQL per request
  Every fallback = a bug waiting to happen
Proposed: Write State at Webhook Time Simple
WRITE PATH (webhook handlers do MORE work):
  subscription.created -->
    user_subscription (same as before)
    + UPDATE user_info SET tier = derive_tier(plan, status)
    + UPDATE user_info SET subscription_state = status.user_status

  invoice.payment_succeeded -->
    stripe_subscription_period (same as before)
    + if billing_reason != 'subscription_create' (not trial $0):
        UPDATE user_info SET ever_paid = TRUE

  subscription.updated -->
    user_subscription (same as before)
    + UPDATE user_info SET tier = derive_tier(plan, new_status)
    + UPDATE user_info SET subscription_state = new_status.user_status

  subscription.deleted -->
    user_subscription (same as before)
    + UPDATE user_info SET tier = FREE (if no other active sub)
    + UPDATE user_info SET subscription_state = INACTIVE

READ PATH (every API request):
  GET /user/profile -->
    SELECT tier, ever_paid, subscription_state FROM user_info
    Total: ~1ms, single row read
    No fallbacks, no edge cases, no JOINs

C. Read Path Simplification repositories.py
New helper (replaces 3 complex methods):

async def update_user_materialized_state(
    session, user_hash,
    tier=None, ever_paid=None,
    subscription_state=None
):
    updates = {}
    if tier is not None:
        updates['tier'] = tier.value
    if ever_paid is not None:
        updates['ever_paid'] = ever_paid
    if subscription_state is not None:
        updates['subscription_state'] = state
    if updates:
        await session.execute(
            update(UserInfo)
            .where(UserInfo.user_hash == user_hash)
            .values(**updates)
        )

async def get_tier(self, user_hash):
    stmt = select(UserInfo.tier).where(
        UserInfo.user_hash == user_hash
    )
    tier = (await self.session.execute(
        stmt)).scalar_one_or_none()
    return UserTier(tier) if tier else UserTier.FREE

5. Team Plan Edge Case: Member Tier Updates
TEAM PLAN: tier is inherited by ALL team members, not just admin

Current get_tier() handles this via Q2/Q4:
  JOIN team_member + team + user_subscription + stripe_subscription_period
  If user is ACTIVE team member of a team with active subscription --> PRO

With materialized state, team member tier updates need to happen at:

  1. subscription.created (team plan)
     Admin only - create team + add admin as member
     UPDATE user_info SET tier=PRO WHERE user_hash = admin_hash

  2. Team member joins (via invitation acceptance)
     New member joins active team
     UPDATE user_info SET tier=PRO WHERE user_id = new_member_user_id
     // Add this to team member join handler

  3. subscription.deleted (team plan)
     Team deactivated, all members lose tier
     UPDATE user_info SET tier=FREE
     WHERE user_id IN (SELECT user_id FROM team_member WHERE team_id = X)
     // Unless member has their own personal PRO sub!
     ...AND user_hash NOT IN (SELECT user_hash FROM user_subscription
         WHERE plan='pro' AND cancel_at IS NULL AND gw_subscription_status='active')

  4. Team member removed (admin removes member)
     Same as #3 but for single user

  5. invoice.payment_failed (team plan)
     UPDATE user_info SET subscription_state=PAYMENT_FAILED
     WHERE user_id IN (SELECT user_id FROM team_member WHERE team_id = X AND role = 'admin')
     // Only admin sees payment_failed, members filtered in router

This is the ONLY added complexity in the write path.
But it replaces the PERMANENT 5-table JOIN in every read.
Team operations are infrequent; reads happen on EVERY request.
6. Safe Migration Strategy (Zero Downtime)
PHASE 1: Dual Write (webhook writes + old reads)
  - Add migration: ensure tier, ever_paid columns are usable
  - Add subscription_state column to user_info
  - Webhook handlers: start writing tier/ever_paid/subscription_state
  - Read path: STILL uses old get_tier()/has_ever_paid() queries
  - Zero risk: writes are additive, reads unchanged

PHASE 2: Backfill
  - Run script: for each user, compute get_tier() + has_ever_paid()
  - Write results to user_info columns
  - Now ALL users have materialized state

PHASE 3: Shadow Read (validate)
  - Read path: read BOTH old queries AND new columns
  - Log any mismatches to monitoring (NewRelic/debuguy)
  - Run for 1-2 weeks to verify consistency
  - Catches any webhook handler gaps

PHASE 4: Switch Reads
  - Read path: switch to reading columns directly
  - Keep old query methods available but unused
  - Instant 350ms+ savings per profile request

PHASE 5: Cleanup
  - Delete get_tier() (150 lines)
  - Delete has_ever_paid() (130 lines)
  - Delete get_user_subscription_state() (130 lines)
  - Delete get_user_subscription_state_by_plan() (80 lines)
  - Delete fallback queries, trial_extension logic
  - Remove index migration for get_tier query optimization
  - ~490 lines of complex SQL deleted
  - No more fallback bugs like PR #1064

8. Safety Net: Nightly Reconciliation Job
Even with materialized state, add a reconciliation cron as safety net:

async def reconcile_user_tiers():
    """Run nightly. Fixes any drift between Stripe truth and our DB."""
    
    # 1. Get all users with active subscriptions in Stripe
    stripe_active = stripe.Subscription.list(status='active', limit=100)
    
    # 2. For each, verify our DB matches
    for sub in stripe_active:
        user_hash = sub.metadata.get('user_hash')
        expected_tier = USER_TIER_MAPPING[plan]
        actual_tier = db.query(user_info.tier).filter(user_hash=user_hash)
        if actual_tier != expected_tier:
            log_alert(f"DRIFT: {user_hash} has {actual_tier}, should be {expected_tier}")
            db.update(user_info, tier=expected_tier)
    
    # 3. Check for users marked PRO but no active Stripe sub
    pro_users = db.query(user_info).filter(tier='pro')
    for user in pro_users:
        if not has_active_stripe_sub(user):
            log_alert(f"STALE PRO: {user.user_hash}")
            db.update(user_info, tier='free')

This catches: missed webhooks, failed event processing, manual Stripe changes,
any bugs in the write path. Runs once/day, low cost, high confidence.
