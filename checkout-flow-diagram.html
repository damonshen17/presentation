How tier & everPaid are Computed Runtime Query
T
get_tier(user_hash) runs UNION ALL of 4 queries:
Q1: Personal sub + valid period (charging_time NOT NULL, period covers now)
Q2: Team sub + valid period (same + team member active)
Q3: Personal sub fallback (no period record yet - webhook delay gap)
Q4: Team sub fallback (same for team)
Returns PRO if any match, else FREE
E
has_ever_paid(user_hash) checks:
- Personal sub with charging_time (excl. trial upgrades)
- Team sub where user was active member during paid period
- Fallback: active gw_subscription_status without period record

3. All Stripe Webhook Events Handled
  • checkout.session.completed Updates stripe_checkout_session state → COMPLETE. No subscription/tier changes.
  • checkout.session.expired Updates stripe_checkout_session state → EXPIRED. Guards against out-of-order events.
  • customer.subscription.created Creates user_subscription row. For team plan: creates team + team_member. Dedup via existing active sub check.
  • customer.subscription.updated Expires old user_subscription, creates new one with updated plan/recurrence. Handles trial→active, month→year upgrades.
  • customer.subscription.deleted Sets cancel_at + expiry_time on user_subscription, status → canceled. Team: soft-deletes team + members.
  • invoice.payment_succeeded Upserts stripe_subscription_period with charging_time. Handles subscription_create / subscription_cycle / subscription_update billing reasons.
  • invoice.payment_failed Upserts stripe_subscription_period with payment_failed_count++. Tracks payment_latest_failed_at.
  • price.created / price.updated Syncs stripe_subscription_price table (price → plan/recurrence mapping). Manages visible_to_user and active flags.

5. Identified Sync Issues & Risks
Sync / Delay Problems Issues
1. Webhook-Only Architecture = Single Point of Failure
If a webhook fails or is delayed (Stripe outage, network issue, server error), the user's tier/subscription state becomes stale. There is no retry mechanism (TODO in code), no reconciliation job, and no way for the user to manually trigger a resync. Failed events stay in FAILED state forever.
2. Race Condition: Polling Window vs Webhook Arrival
After checkout, frontend polls for 10 seconds (1s × 10). But the critical customer.subscription.created webhook can take 0-30+ seconds to arrive and process. If it arrives after polling ends, the user sees themselves as FREE until they manually refresh. The fallback queries (Q3/Q4) mitigate this but only if subscription.created webhook arrives within the poll window.
3. Out-of-Order Webhook Events
Stripe doesn't guarantee event ordering. Code handles some cases (checkout session state guards, existing active subscription check), but subscription.updated arriving before subscription.created will fail with "subscription not found". The invoice.payment_succeeded with billing_reason=subscription_update relies on finding a previous period that may not exist yet.
4. get_tier() Complexity = Performance & Correctness Risk
get_tier() runs a UNION ALL of 4 subqueries with JOINs across 5 tables on every profile request. Currently ~150ms but fragile: any new plan type, edge case, or data inconsistency requires updating complex SQL. The fallback queries (Q3/Q4) exist specifically to handle webhook delays, adding permanent complexity for a transient problem.
5. Dead Columns: user_info.tier & ever_paid_for_pro
Legacy columns still exist in the DB and model but are never updated. Any code accidentally reading these (admin panel, reporting queries, data exports) would get stale/wrong values. Risk of confusion for new developers.
Improvement Recommendations Solutions
A. Add Webhook Retry + Dead Letter Queue
Implement the TODO: retry failed events (3 attempts with exponential backoff). After max retries, move to a dead letter queue with alerting. This prevents permanent state desync from transient failures.
B. Stripe API Fallback on Cache Miss
When get_tier() returns FREE for a user who just completed checkout, add an on-demand Stripe API check: stripe.Subscription.list(customer=cust_id, status='active'). Only trigger this when checkout session is COMPLETE but no active subscription exists. This closes the race window completely.
C. Simplify tier to a Materialized Column
Instead of computing tier on every request with 4-query UNION ALL, update user_info.tier directly when webhooks process. The column already exists! This eliminates: (1) 150ms query overhead on every profile call, (2) fallback query complexity, (3) edge cases from eventual consistency. Webhook handlers already know the correct tier - just write it.
D. Add Periodic Reconciliation Job
Run a nightly cron that compares Stripe subscription states with local DB. For any mismatches, fetch the latest from Stripe API and update DB. This catches any missed/failed webhooks and provides a safety net for the webhook-only architecture.
E. Extend Polling or Use Server-Sent Events
Current 10-poll window is too tight. Options: (1) Increase to 30 polls / 30s, (2) Switch to SSE/WebSocket that pushes tier change when webhook processes, (3) Use checkout session ID to poll a dedicated endpoint that checks if subscription.created webhook has been received.
F. Clean Up Dead Columns
Remove user_info.tier and user_info.ever_paid_for_pro columns (or repurpose them per option C). This eliminates confusion and prevents accidental reads of stale data.
6. Recommended Simplified Architecture
CURRENT (Complex):
  User Request → get_tier() → 4-query UNION ALL across 5 tables (~150ms)
  Webhook → update user_subscription + stripe_subscription_period (tier computed at read time)
  Frontend → poll 10x hoping webhook arrived

Problems: Expensive reads, race conditions, no retry, fallback query complexity

PROPOSED (Simple):
  User Request → read user_info.tier directly (~1ms)
  Webhook → update user_subscription + stripe_subscription_period + UPDATE user_info SET tier=X
  Webhook fail → retry 3x → dead letter queue + alert
  Nightly cron → reconcile Stripe vs DB, fix any drift
  Frontend → poll with Stripe API fallback for checkout flow

Benefits: O(1) tier reads, no race window, self-healing, simpler code
